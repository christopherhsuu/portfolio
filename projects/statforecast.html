StatForecast

A forecasting tool that uses historical sabermetrics and machine learning to estimate future player performance. This page links to the live app and the code repository.

Methodology & Execution

Overview

PitchSequence is a sabermetric machine-learning pipeline and interactive Streamlit app that recommends the next pitch a pitcher should throw in a given situation. It blends Statcast data, batter archetypes, pitcher arsenals, and a run-value prediction model to simulate how decisions impact expected runs.

The system integrates data processing, feature engineering, model training, and UI design to create an explainable recommender that mirrors real-world pitch-calling logic.

1. Data Preparation

Folder Purpose
data/raw/ Raw Statcast dumps (statcast_2025.csv, name mapping files)
data/processed/ Cleaned and feature-engineered Parquet files (cleaned.parquet, features.parquet, runvalue.parquet)
pitcher_assets/pitcher_arsenals.csv Inferred pitcher arsenals derived from Statcast
  • Data ingestion begins by placing raw CSVs in data/raw/ and running src/preprocess.py, which:
    • Parses timestamps and cleans inconsistent or missing rows.
    • Normalizes pitch type labels (FF, SL, CH, etc.).
    • Ensures batter and pitcher names align across datasets.
    • Drops invalid rows lacking pitch type or outcome information.
  • Next, src/feature_engineering.py computes play-level and aggregate features:
    • Maps categorical pitch locations ("Top-In", "Low-Away") to numeric plate coordinates (plate_x, plate_z).
    • Calculates last_pitch_speed_delta and contextual deltas.
    • Aggregates batter vs. pitch_type statistics such as:
      • batter_pitchtype_woba
      • batter_pitchtype_whiff_rate
      • batter_pitchtype_run_value
  • Data cleaning & filtering:
    • Missing numeric values are imputed with contextual defaults (e.g., 0 for missing rates).
    • Unrealistic outliers (e.g., release speeds >105 mph) are winsorized.
    • Pitchers/batters with insufficient sample sizes are filtered out to stabilize estimates.
    • Only recent seasons (1–3 years) are used for modeling to avoid historical drift.
    • All processed outputs are saved to data/processed/ for efficient re-use during model training.

2. Feature Engineering

  • Core features:
    • Count & base state: balls, strikes, outs_when_up, on_1b, on_2b, on_3b
    • Pitch data: pitch_type, release_speed, release_spin_rate
    • Plate location: plate_x, plate_z, zone
    • Temporal context: last_pitch_type, last_pitch_result, last_pitch_speed_delta
    • Batter matchup metrics: batter_pitchtype_woba, batter_pitchtype_whiff_rate, batter_pitchtype_run_value
  • Transformations:
    • Categorical features are one-hot or target-encoded in the model pipeline.
    • Numeric variables are scaled as needed.
    • Interaction terms such as count × pitch_type capture strategic context.
  • Rationale:

    The feature space merges physical characteristics (pitch speed, location) with situational and matchup information, enabling the model to predict not just what is thrown, but what should be thrown to minimize run expectancy.

3. Model Training

  • The training logic resides in src/model_train.py, producing a serialized artifact at artifacts/runvalue_model.pkl.
  • Pipeline steps:
    1. Load preprocessed data from data/processed/features.parquet and run-value targets from runvalue.parquet.
    2. Split data into train/validation/test sets using a time-based split to prevent data leakage.
    3. Construct a scikit-learn pipeline for:
      • Categorical encoding
      • Missing-value imputation
      • Scaling
      • Regression model training
    4. Train candidate regressors such as:
      • RandomForestRegressor
      • GradientBoostingRegressor
    5. Optimize hyperparameters via grid or randomized search.
    6. Evaluate with MSE, MAE, and explained variance, selecting the model with the lowest error on validation data.
    7. Save the fitted model using joblib.dump() or pickle for deployment.
  • Notes:
    • Version alignment for scikit-learn is crucial to avoid unpickling errors.
    • For long-term maintenance, pin dependencies in requirements.txt.

4. Evaluation

  • Evaluation focuses on both numerical accuracy and strategic validity:
  • Metrics:
    • Mean Squared Error (MSE)
    • Mean Absolute Error (MAE)
    • Explained Variance
  • Decision-support quality:
    • Tests whether the model’s ranked pitch recommendations correlate with lower realized run values.
  • Diagnostics:
    • Residual plots
    • Feature importance charts
    • Partial dependence plots
  • A time-aware cross-validation (sliding-window approach) ensures robustness against season-to-season variation.

Methodology & Execution

Overview

StatForecast is an interpretable machine-learning pipeline designed to forecast next-season offensive statistics for MLB hitters. The project demonstrates how trajectory-aware features—such as lagged season stats, rolling means, and year-over-year deltas—can improve predictive performance beyond naïve single-season models. Built with Python, pandas, and scikit-learn, it culminates in an interactive Streamlit app that allows users to explore player histories and generate live projections.

1. Data Preparation

  • All data originated from a season-by-season hitter CSV (e.g., fangraphs_hitters_2010_2024.csv) formatted in the Fangraphs or Baseball-Reference style. Each row represented a single player-season, including identifiers (Name, Season) and standard stats (PA, HR, AVG, OBP, SLG, ISO, wOBA, BB%, K%).
  • Data ingestion and cleaning were handled by prediction.py::load_data, which:
    1. Loaded the CSV via pandas.read_csv.
    2. Normalized inconsistent columns (Player → Name, Year → Season).
    3. Converted percent strings (e.g., "12.3%") to float ratios for BB% and K%.
    4. Filtered out players with fewer than 200 plate appearances to reduce variance.
    5. Sorted and reindexed the DataFrame by Name and Season.
    6. Validated all required columns, raising a clear ValueError if any were missing.
  • Missing lag features were dropped at training time, while rookies (with no prior seasons) had missing lag values imputed with zeros during prediction. Outlier clipping was intentionally minimal to preserve natural variance in the data.
  • To maintain reproducibility, datasets were excluded from version control (.gitignore includes *.csv), while trained models were either stored externally or downloaded dynamically via a MODEL_URL.

2. Feature Engineering

Feature creation was the core of this project, executed in prediction.py::build_features.

  • Base predictors: ['Age', 'PA', 'BB%', 'K%', 'HR', 'ISO', 'wOBA']
  • Targets (multi-output): ['AVG', 'SLG', 'OBP', 'OPS', 'wRC+', 'HR', 'ISO', 'wOBA']
  • Key engineered feature groups:
    • Lag Features: lag1_, lag2_, lag3_ for each stat and for PA and Age to capture 1–3-year performance memory.
    • Rolling Means: roll3_<stat> computed as a three-year rolling mean (shifted by one season to prevent data leakage).
    • Delta Features: delta1_ and delta2_ track season-to-season directional changes.
    • Averaged Deltas: avg_delta_<stat> smooths noisy variation.
    • Career Momentum: career_change3_<stat> measures the difference between the current and three-seasons-ago performance.
    • Age Interactions: age_x_avg_delta_<stat> integrates age-related improvement or decline rates.
  • These features model both persistence and trajectory, embedding sabermetric intuition: players don’t just perform—they trend. A 24-year-old improving slugger, for instance, carries a very different predictive signal than a 36-year-old in decline.

3. Model Training

  • The model pipeline lives in prediction.py::train_model, which executes:
    1. Feature generation via build_features(df, lags=3).
    2. Assembly of predictor (X) and target (Y) matrices using _assemble_matrix, dropping rows missing necessary inputs or next-season targets.
    3. Saving of feature_columns to ensure consistent reindexing at inference time.
    4. Splitting data with train_test_split(X, Y, test_size=0.15, random_state=42).
    5. Fitting a MultiOutputRegressor(RandomForestRegressor(n_estimators=200)), chosen for its interpretability and robustness with heterogeneous baseball stats.
    6. Computing RMSE (Root Mean Squared Error) across all targets using NumPy.
    7. Saving the serialized payload—model + metadata—via joblib.dump({'model': model, 'metadata': metadata}, save_path).
  • Saved metadata includes:
    • List of exact training-time feature_columns
    • Test/train sample counts
    • Per-target RMSE values

4. Evaluation & Diagnostics

  • Performance was assessed primarily using RMSE, giving intuitive, unit-preserving error magnitudes (e.g., deviations in AVG points or home runs).
  • Recommended exploratory diagnostics included:
    • Residual plots per target (predicted vs. actual).
    • Error distributions and outlier review (e.g., rookies, injury returns).
    • Uncertainty heuristics by computing per-sample prediction variance across individual forest trees via model.estimators_.
  • Though the initial version used a single hold-out split, the framework is cross-validation-ready: hyperparameters (n_estimators, max_depth, min_samples_leaf) can be tuned through grid or randomized search optimizing multi-output RMSE.

5. Deployment

  • Deployment was implemented in streamlit_app.py, providing an intuitive interface for analysts, fans, or researchers.
  • Runtime behavior:
    • Model Loading: Loads a local models/rf_multioutput.joblib, or downloads it dynamically if a MODEL_URL secret is configured (useful for Streamlit Cloud).
    • Data Upload & Normalization: User-uploaded CSVs are validated through load_data, ensuring consistency.
    • Interactive Player Search: A searchable dropdown built from df['Name'].dropna().unique() allows player selection.
    • Prediction Execution: predict_player(player_name, df, model) re-builds features, reindexes columns to match feature_columns, fills missing inputs with zeros, and outputs next-season forecasts.
    • Visualization: Player history and projection plots are rendered inline with Streamlit’s plotting API.
  • This design keeps deployment frictionless: model binaries stay external (via Git LFS or runtime download), and retraining can occur directly in the app.

8. Design Notes

  • Consistency is enforced throughout by saving the model’s feature_columns.
  • The Streamlit selectbox relies on exact string matches for player names; normalization ensures lookup stability.
  • Large binaries should never be committed directly—use Git LFS or external hosting to keep history clean and deployment lightweight.
